C++ 헥사고날(포트-어댑터) CLI · 일정(Event)·할일(Todo)·목표(Goal) · 식물 메타포 네이밍 · 실행물 gaia

개요 / 오리엔테이션

C++ 헥사고날(포트-어댑터) CLI 도구. 일정(Event)·할일(Todo)·목표(Goal) 세 도메인을 SQLite 에 저장하고 관리한다. 식물 메타포 네이밍을 쓴다.

진입점

  • 실행 진입점: src/glass/main.cpp (main) — 유일한 Composition Root. 빌드 산출물 gaia.
  • 논리 진입점: src/stem/*UseCase::execute() 16개 (Event/Todo/Goal 의 CRUD·List·Show + Dashboard).

모듈별 동작 서술. 관통 원칙: 시간/타임존 계산은 엣지에서, 내부는 UTC instant·달력날짜만. 충돌검사·반복 전개·목표 누적.

호출그래프

clang/LLVM 함수 호출 그래프. 헥사고날 경계의 정적 증거(stem→어댑터 직접엣지 0, 전부 포트 경유 virtual). 오버뷰 + 드릴다운 17장.

관통 설계 원칙: 벽시계·로컬 타임존·달력 계산은 모두 엣지(glass/leaves)에서, 내부 도메인/저장소는 UTC instant(sys_seconds)와 순수 달력 날짜(sys_days)만 다룬다.

아키텍처 / 의존

식물 메타포로 네이밍된 헥사고날(포트-어댑터) 아키텍처의 CLI 애플리케이션이다. 핵심 도메인/유스케이스(planning_core)가 인터페이스(stomata 포트)에만 의존하고, 구체 기술(SQLite, spdlog, toml++, CLI11)은 별도 어댑터 라이브러리로 분리되어 의존성이 안쪽으로만 향한다. 실행 파일 gaia(src/glass/main.cpp)가 유일한 컴포지션 루트로서 어댑터를 조립한다.

진입점

main() — src/glass/main.cpp. 유일 실행 진입점. 모든 어댑터(roots/rings/climate/leaves)와 도메인/유스케이스(seed/stem)를 직접 include 하여 조립하는 컴포지션 루트. gaia 실행 파일로 빌드. main.cpp 는 어댑터 5종, 도메인 헤더, 17개 유스케이스, 4개 command DTO 를 모두 include 한다.

내부 의존성 방향

어댑터(roots/rings/climate/leaves)는 포트(stomata)를 구현하고, 유스케이스(stem)는 포트에만 의존한다. 따라서 도메인/유스케이스는 구체 기술을 모른다(의존성 역전). 진입점 glass만 어댑터와 코어를 동시에 알고 조립한다. 순환 의존 없음 — 모든 화살표가 진입점 → 어댑터/유스케이스 → 포트 → 도메인 방향의 단방향이다.

관찰 메모

  • 의존성 역전이 일관되게 적용됨: 4개 어댑터가 각각 stomata 포트를 implements 하며, stem 유스케이스 헤더에는 구체 어댑터 include 가 전혀 없다.
  • 포트 라이브러리 분리 없음: stomata/seed/stem이 모두 planning_core 한 타겟에 묶여 있다. 어댑터들은 외부 라이브러리를 PRIVATE 으로 캡슐화한다.
  • stem/queries/ 디렉토리는 존재하나 비어 있음 (조회용 DTO 미구현).
  • 외부 헤더 uuid.h(stduuid)가 seed 엔티티 헤더에 직접 노출된다 — 도메인이 id 생성용 vendor 타입에 헤더 수준으로 의존하는 지점.

타입 / 관계

헥사고날(포트-어댑터) 구조로, 식물 메타포 디렉터리가 레이어에 대응한다.

디렉터리네임스페이스역할

1. 도메인 타입 (seed/)

엔티티 (Aggregate Root)

  • Goal — 목표값 대비 누적값으로 달성률. Id = uuids::uuid. 멤버: id_, name_, targetValue_, currentValue_, unit_, periodStart_, periodEnd_. 행위: incrementCounter, rename, updateTarget, updatePeriod, progressRatio. 불변식 위반 시 std::invalid_argument.
  • Todo — 할 일. 멤버: id_, title_, done_, priority_, tags_, due_. 행위: markDone, rename, updatePriority, addTag(멱등), removeTag, setDueDate.
  • Event — 일정. 멤버: id_, title_, range_(TimeRange), recurrence_(optional<RecurrenceRule>). 행위: reschedule, rename, setRecurrence.

값 객체

  • TimeRange — 불변 시간 구간(반열린 [start, end)). overlaps()로 겹침 판정. start > end 위반 시 예외.
  • RecurrenceRule (struct) — frequency, until(optional<sys_seconds>).
  • Conflict (struct) — existingEventId, existingTitle, existingRange.

열거형 / 도메인 서비스·포트

  • Priority: HIGH, MEDIUM, LOW · RecurrenceFrequency: Daily, Weekly, Monthly, Yearly
  • ConflictDetector — detect(candidate, existingOverlapping) → optional<Conflict>.
  • IdGenerator (추상) — next() → uuids::uuid. 도메인에 위치한 포트. · StdUuidGenerator — 구현(stduuid).

2. 포트 인터페이스 (stomata/)

  • GoalRepository — findById, findByName, findAll, save, update, remove.
  • TodoRepository — findById, findByDueDate, findOverdue, findByTag, findByPriority, findAll, save, update, remove.
  • EventRepository — findById, findOverlapping, findInRange, findAll, save, update, remove.
  • ConfigLoader — logConfig() → LogConfig, dbPath(). 중첩 struct LogConfig.
  • ConflictPrompter — promptOnConflict(Conflict) → Choice. 중첩 enum Choice(ADD_ANYWAY, CANCEL).
  • Logger — debug, info, warn, error, audit(action, detail).

관찰 노트

  • 상속/구현은 모두 인터페이스 실현 1단계뿐이며, 엔티티 간 상속은 없다(합성만). Goal/Todo/Event 세 도메인이 CRUD 대칭 구조다.
  • IdGenerator 포트는 다른 포트와 달리 domain 네임스페이스에 위치(도메인이 식별자 생성을 직접 추상화).
  • 유스케이스는 구현체가 아닌 포트 추상에만 의존. 시간/타임존 계산은 도메인에 두지 않고 엣지에서 UTC instant 로 변환해 Command/Query 로 주입한다.

플로우 — 주요 시나리오의 호출/데이터 흐름

이 CLI 는 단일 진입점 main(glass) 에서 CLI11 로 인자를 파싱하고, "Composition Root" 에서 모든 어댑터·도메인·유스케이스를 한 번에 조립한 뒤, 어느 서브커맨드가 파싱됐는지에 따라 해당 유스케이스(stem)를 호출하는 헥사고날 구조다. 유스케이스는 포트 인터페이스(stomata)를 통해서만 저장소(roots, SQLite)와 도메인(seed)에 접근하고, 결과를 main 으로 돌려주면 main 이 leaves 의 포맷 함수로 콘솔에 출력한다.

공통 부트스트랩: main 은 먼저 모든 옵션·서브커맨드를 CLI11 에 등록하고 CLI11_PARSE로 파싱한다. init이 파싱된 경우에만 Composition Root 진입 전에 분기(시나리오 3). 그 외에는 try 블록에서 TomlConfigLoader로 설정을 읽고, SpdlogLogger·SQLite::Database(WAL/foreign_keys PRAGMA)·MigrationRunner.run으로 스키마를 올린 뒤, 세 저장소·ConflictDetector·StdUuidGenerator·CliConflictPrompter를 만들고 생성자 주입으로 16개 유스케이스를 인스턴스화한다. 그다음 거대한 if/else 체인이 디스패치한다. 예외는 맨 바깥 catch 가 오류: ...를 stderr 로 찍고 종료코드 1.

시나리오 1 · event add — 충돌 검사와 사용자 확인이 있는 쓰기 경로

대표성: 저장 전에 도메인 검증·저장소 조회·도메인 충돌 판정·대화형 프롬프트를 모두 거치는, 가장 흐름이 긴 쓰기 경로.

main의 eventAdd 분기는 파싱된 문자열 옵션을 leaves 의 parseDateTime/makeRule로 변환해 CreateEventCommand를 채우고 createEvent.execute(cmd)를 호출한다. CreateEventUseCase::execute는 먼저 domain::TimeRange 생성자로 시간 구간을 검증한다(잘못된 구간이면 invalid_argument가 올라가 catch 로). 이어 events_.findOverlapping(range)로 겹칠 후보를 조회하는데, SQLite 저장소는 start_ts < candEnd AND COALESCE(end_ts, start_ts) > candStart SELECT 로 후보 행을 모아 도메인 Event로 복원한다. 유스케이스는 후보를 detector_.detect(range, overlapping)에 넘기고, ConflictDetector는 각 후보의 TimeRange::overlaps를 확인해 첫 충돌을 Conflict로 반환한다. 충돌이 있으면 prompter_.promptOnConflict가 호출되어 "그래도 추가할까요? [y/N]"를 출력하고 stdin 한 줄을 읽는다. y/Y/yes가 아니거나 EOF 이면 CANCEL을 반환하고, 유스케이스는 감사 로그 후 cancelledByUser=true Result 를 돌려줘 main 이 "취소되었습니다"를 출력한다. 진행이면 idGen_.next()로 UUID 를 발급해 domain::Event를 만들고 events_.save(event)가 INSERT 를 실행, 감사 로그 후 Result 를 돌려줘 main 이 "Event ... 추가 완료"를 출력한다.

시나리오 2 · 서브커맨드 없음 — 대시보드(두 저장소 동시 조회)

시나리오 3 · init — Composition Root 이전의 설정 부트스트랩

대표성: 설정을 "읽는" 다른 모든 명령과 달리 설정을 "쓰는" 명령이라, DB·로거·저장소 조립 자체를 건너뛰고 파싱 직후 따로 처리되는 예외적 흐름.

CLI11_PARSE 직후 main은 initCmd->parsed()를 가장 먼저 검사한다. 참이면 --db/--log 경로를 renderDefaultConfig(initDb, initLog)에 넘기는데, climate 어댑터는 toml++ 로 [database].path와 [log] 섹션(level=INFO, audit=true, rotation=none, retention 일수 등 기본값)을 만들어 TOML 문자열로 직렬화해 반환한다. main 은 그 문자열을 --config 경로에 trunc(항상 덮어쓰기) 모드 ofstream으로 기록하고, 열 수 없으면 runtime_error를 던진다(로컬 try/catch 가 받아 "오류: ..."를 stderr 로 출력, 1 반환). 성공하면 "설정 파일 생성: ..."을 출력하고 즉시 return 0하므로 SQLite 연결이나 마이그레이션은 전혀 일어나지 않는다.

로직요약 — 일정/할일/목표 관리 도구

이벤트(일정), 할일(Todo), 목표(Goal) 세 도메인을 다루는 CLI. 헥사고날 구조, 식물 메타포 네이밍. 일관되게 관통하는 설계 원칙은 "벽시계·로컬 타임존·달력 계산은 모두 엣지(glass/leaves)에서 하고, 내부 도메인/저장소는 UTC instant(sys_seconds)와 순수 달력 날짜(sys_days)만 다룬다"는 것이다.

glass — 진입점 / Composition Root

프로그램의 유일한 main. CLI11로 서브커맨드 트리를 정의하고 인자를 Command/Query 로 변환해 호출한다. 책임 셋:

  • 의존성 조립: 설정 로드 → 로거 → SQLite 연결 → 마이그레이션 → 저장소·서비스·유스케이스 생성·주입. 구체 구현을 아는 유일한 자리.
  • 벽시계 의존 정책: localTodayDate()로 로컬 "오늘"을 읽는다. 의도적으로 Composition Root 에만 둔 부수 효과.
  • 부분 수정 플래그 충돌 검증: --end/--clear-end, --all-day/--no-all-day, --repeat/--no-repeat, --due/--clear-due, --tag/--clear-tags 상호 배타. 목표 기간 변경은 --from/--to를 반드시 함께.

부수 효과: 표준출력/표준에러로 결과·오류 출력, 모든 도메인 예외를 잡아 "오류: ..." 후 종료코드 1. init은 설정을 쓰는 동작이라 Composition Root 진입 전에 따로 처리(항상 trunc로 생성). --config는 항상 필수.

seed — 도메인 엔티티와 순수 규칙

외부 의존이 없는 핵심 비즈니스 객체들. 모두 생성자에서 불변식 검증.

  • Event: 제목·시간구간·선택 반복규칙. 빈 제목이면 예외.
  • Todo: 제목·우선순위·태그·마감일·완료여부. 태그 추가는 멱등.
  • Goal: 목표값 대비 누적값으로 달성률. targetValue<=0 또는 periodStart>periodEnd면 예외. 달성률 상한 없음(초과 시 1.0 초과 가능). 복원용 누적값 생성자 별도.
  • TimeRange: 시작·(선택)종료·종일. start>end면 예외. 핵심은 overlaps() — 종료시각 없으면 시작을 종료로 간주(effectiveEnd)하고 반열린구간 겹침 판정.
  • ConflictDetector: 겹치는 후보 목록을 받아 첫 충돌 반환. 겹침 후보 조회는 저장소가, 판정만 이 클래스가.
  • StdUuidGenerator: random_device 시드로 UUID v4 생성.

stem — 유스케이스 (애플리케이션 서비스)

각 유스케이스는 포트만 의존, 도메인 객체를 조립·조작해 저장소에 위임. 공통: 끝에 logger.audit(...). 로직이 실린 곳:

  • CreateEventUseCase / UpdateEventUseCase: 저장 전 시간 충돌 검사. findOverlapping → ConflictDetector 판정 → 충돌 시 ConflictPrompter로 강행/취소. Update 는 자기 자신 제외, 시간이 바뀐 경우에만 검사. 취소 시 저장 안 하고 cancelledByUser 반환.
  • ListEventsUseCase: 가장 무거운 곳. findAll 후 비반복은 윈도우 겹침 필터, 반복은 [winStart,winEnd) occurrence 전개. 빈도만큼 전진하며 월/년 반복에서 존재하지 않는 날은 그 달 마지막 날로 클램프. 무한루프 방지 kSafetyCap=200000. 시작시각 오름차순 정렬.
  • CreateGoalUseCase: 같은 이름 목표 있으면 거부(이름 유일성).
  • LogGoalUseCase / ShowGoalUseCase: findByName 직접 조회. 진행 기록은 기간 종료와 무관하게 누적.
  • ListTodosUseCase: 전부 가져와 인메모리로 오늘 마감/기한초과/태그/우선순위 필터 AND 결합. "기한초과"는 미완료이면서 마감일이 기준일보다 과거.
  • UpdateTodoUseCase: 부분 수정. 태그 주어지면 기존 전부 제거 후 전체 교체.
  • ShowDashboardUseCase: 오늘 이벤트 수 + 기한초과 Todo 수 집계. 반복 인스턴스 카운트 미반영(TODO) — 이벤트 카운트는 비반복 기준.

Update/Delete/Done/Log 계열은 대상이 없으면 std::out_of_range(엣지에서 잡아 오류 출력).

stomata — 포트 인터페이스

순수 추상 인터페이스 모음: EventRepository, TodoRepository, GoalRepository, Logger, ConfigLoader, ConflictPrompter. 테스트 대역 주입·어댑터 교체 가능. 저장소 포트는 "로컬 달력 경계 계산은 엣지가, UTC instant 로 넘긴다" 계약을 주석으로 못박는다.

roots — SQLite 영속성 어댑터

  • SqliteEventRepository: 시각은 epoch 초(int64), UUID 는 문자열. 핵심은 findOverlapping/findInRange의 겹침 SQL — start_ts < ? AND COALESCE(end_ts, start_ts) > ?로 반열린구간 규칙을 그대로 구현. 반복규칙은 recur_freq/recur_until 컬럼.
  • SqliteGoalRepository: 날짜는 epoch days. findByName 이름 조회 지원(Create 유일성, Log/Show 직접 조회 근거).
  • SqliteTodoRepository: 동형 Todo CRUD.
  • MigrationRunner: schema_version 테이블 생성, 마이그레이션 디렉토리(TERRARIUM_MIGRATIONS_DIR)의 .sql을 버전 순 정렬·적용. 적용된 버전은 건너뛰고 각 마이그레이션을 트랜잭션으로 감싸 실패 시 롤백.

climate — 설정

  • TomlConfigLoader: TOML 을 읽어 DB 경로(필수)·로그 설정 제공. 파일 없음·파싱 오류를 모두 runtime_error로 통일.
  • DefaultConfig (renderDefaultConfig): init이 쓸 기본 설정 TOML 문자열 생성. toml++가 키를 정렬 직렬화하므로 출력 순서는 비보장(주석 명시).

rings — 로깅

spdlog 기반 Logger 포트 구현. 회전 전략(none/daily/size)을 골라 파일 싱크를 만들고, 디버그/감사 로그를 분리(separateDebugAudit)할 수 있다. 파일 싱크 생성 실패 시 예외를 밖으로 던지지 않고 stderr 폴백 — 로깅 실패가 앱을 죽이지 않게. audit 꺼져 있으면 감사 로그 무시.

leaves — CLI 입출력 어댑터

  • CliFormat: 문자열 파싱(parseDateTime/parseDate/parsePriority/parseFrequency)과 포맷팅, 그리고 로컬↔UTC 변환의 핵심(localCivilToUtc/localMidnightUtc). 로컬 시각을 UTC instant 로, 저장된 UTC 를 다시 로컬로 표시. 진행 막대는 #/- ASCII.
  • CliConflictPrompter: 충돌 시 "[y/N]"로 묻는다. y/Y/yes만 강행, 그 외와 EOF(비대화형)는 안전하게 취소.

전체적 가정과 제약

  • 시간 정책: 입력/표시는 로컬 타임존, 내부 저장은 UTC. 달력 경계는 엣지에서 계산해 UTC instant 로 변환 후 저장소에.
  • 목록/대시보드의 반복 이벤트는 인메모리 findAll 후 전개(성능 최적화 TODO), 대시보드 이벤트 카운트는 반복 인스턴스를 세지 않음.
  • 목표 이름은 유일하며, 목표 진행 기록은 기간 만료를 검사하지 않고 무조건 누적.
  • 비대화형 환경에서 충돌이 나면 입력이 없으므로 항상 취소.

C++ 함수 호출 그래프

이 그래프는 큐레이션본이다. clang/LLVM dot-callgraph 원본에서 프로젝트 planning:: 함수만 남기고 ctor/dtor·사소한 getter·익명 네임스페이스 헬퍼·람다 클로저를 제거했다. 최종 의미함수 84개만 다룬다. stomata(ports)는 순수 추상 인터페이스라 정적 호출그래프에 함수 노드가 0개이며, 모듈 박스는 "포트 seam" 표현용. 간접/가상 호출 엣지는 클래스계층 근사(추정)이며 점선+"est" 라벨이다.

아키텍처 한눈에 (헥사고날)

  • 진입점은 main(glass) 하나. main 이 climate 로 설정을 읽고(TomlConfigLoader::dbPath/logConfig, 없으면 renderDefaultConfig), roots 의 MigrationRunner::run으로 마이그레이션을 돌린 뒤, 16개 유스케이스(stem)를 직접 디스패치.
  • 의존 방향: glass → stem → seed 는 직접 호출. stem → {roots, rings, leaves}는 stomata 인터페이스를 통한 가상 디스패치(추정). 도메인(seed)은 어느 어댑터도 호출 안 함(의존 역전 성립).

모듈 간 호출 관계

  • 직접 호출(실선): glass→stem 16, glass→seed 9, glass→leaves 9, glass→climate 3, glass→roots 1; stem→seed 20; roots→seed 14; seed→seed 3; leaves→leaves 2.
  • 추정 호출(점선, 포트 경유): stem→roots 17, stem→rings 16, stem→leaves 2.
  • 핵심 관찰: stem 은 roots/rings 를 직접 호출하지 않는다 — 전부 추정(포트 경계). 포트-어댑터 의존성 역전이 정적 그래프에 남긴 흔적.

추출: clang/LLVM 17, -emit-llvm → llvm-link → opt -passes=dot-callgraph → llvm-cxxfilt. 대상 src/ 32 TU(벤더·테스트·build 제외). 노드 225·직접엣지 272·virtual 35·진입점 17.
